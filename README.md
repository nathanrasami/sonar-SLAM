# 🚀 Run & analyse — branche `Bruce` (fork stage 4A, Aracati2017 GT-free)

> Bruce-SLAM original adapté au dataset Aracati2017 (cas d'étude doctorante).
> Un seul run à la fois ; ne rien modifier dans le dépôt pendant un run.
> Docs : `ABLATION.md` (protocole complet A/B), `PIEGES.md` (avant tout debug), `TESTS.md` (runs archivés).

## 1. Lancer un run (s'arrête SEUL à la fin du bag, ~45 min)

```bash
SSM=true NSSM=true USBL=false ./run_slam.sh                                # A  = Bruce pur        (attendu ~1.9-2.1 m)
SSM=true NSSM=true USBL=true USBL_GAIN=0 USBL_BACKEND=true ./run_slam.sh   # B′ = A + ancre USBL   (champion, ~1.88 m)
./run_slam.sh                                                              # nu = Bruce «pristine» (filtre USBL front, ~3.4 m — pas la config d'étude)
```

Sorties dans `results/run_aracati_<date>/` (CSV + RViz s'ouvre ; ESPACE dans le terminal = pause du bag).

## 2. Toutes les variables d'environnement

| Var | Défaut | Rôle |
|---|---|---|
| `SSM` / `NSSM` | false / false | modules natifs Bruce : recalage court-terme / fermetures de boucle géométriques |
| `USBL` | true | filtre USBL **front-end** (gain ci-dessous) |
| `USBL_GAIN` | 0.4 | gain du filtre front. **⚠ TOUJOURS 0 si `USBL_BACKEND=true`** (sinon double ancrage → zigzag, ATE ×3, PIEGES §2) |
| `USBL_BACKEND` | false | facteurs USBL robustes dans le graphe (σ2.5 figé dans le yaml) |
| `RATE` | 1.0 | vitesse de rejeu du bag |
| `BAG` | ARACATI_2017_8bits_full.bag | autre bag |

## 3. Analyser

```bash
./analyse.sh run_aracati_<date>                          # chaîne complète de figures
python3 analysis/bilan_run.py results/run_aracati_<date> # bilan 1 image (rapide)
python3 analysis/paper_eval.py results/run_aracati_<date> --label monrun   # chiffres protocole papier (S1/S2/S3, RE)
```

## 4. Lire les résultats

| Sortie | Signification | Attendu (B′ / A) |
|---|---|---|
| console `ATE Umeyama` | erreur de trajectoire après meilleur recalage rigide | ~1.88 / ~1.97 m (méd. multi-runs) |
| console `Cap méd` | résidu de cap vs compas — LA force de cette branche (SSM) | 1.8-3.0° |
| `bilan_run.png` | 3 panneaux : traj+ATE, nuage+NN, cap dans le temps | — |
| `error_over_time.png` | erreur de position : plein = Umeyama, pointillé = ancré départ | — |
| `pointcloud_map.png` + NN | carte (rendu θ optimisé — le rendu compas n'apporte rien ici) | NN ~0.205 m |
| `trajectory.csv` | `theta` = cap SLAM ; `dr_theta` = cap compas intégré ; positions x,y optimisées | — |

# BlueRov SLAM 

Welcome! This repo contains the code for BlueROV sonar-based SLAM. This system uses a source of dead-reckoning, in our case a DVL and IMU with environment observations from an imaging sonar to perform graph-based pose slam using the ISAM2 backend implemented by gtsam. This method employs the scan matching paradigm to align scans using dead-reckoning as an initial guess. Point clouds for ICP are generated by converting sonar images to in-plane clouds. Note that this system is 3DOF and we assume fixed depth motion. 

This is the original work of Jinkun Wang. The documentation, maintenance, and active development are by John McConnell.

# Sensor overview

Our vehicle is documented in this repo https://github.com/jake3991/Argonaut.git. The highlights are the following pieces of hardware. 
- Occulus M750d imaging sonar
- Occulus M1200 imaging sonar (optional)
- Rowe SeaPilot DVL
- Vectornav 100 MEMS IMU
- Bar30 pressure sensor
- KVH-DSP-1760 fiber optic gyroscope (optional)

# Installation

- Check ros distro
- clone this repo into your catkin workspace

# Running "Online"
This will launch the SLAM system, then we will playback the data as if it is happening now. 
- source catkin_ws/devel/setup.bash
- roslaunch bruce_slam slam.launch
- rosbag play your_data.bag

# Running Offline
This runs our offline mode, great for quick testing/tuning of parameters. 
- source catkin_ws/devel/setup.bash
- roslaunch bruce_slam slam.launch file:=path_to_data/your_data.bag

# Configuration
This SLAM system has many parameters, please read the wiki for an explanation of each parameter. However, we highly recommend using the default parameters in the config folder. If you are to tune anything it would be the feature extraction node in feature.yaml.
